Washington, DC Smart Travel Planner

Geospatial Portfolio Project Proposal

Summary

This project is about building a travel planning tool for Washington, DC that feels practical and easy to use. A visitor can choose how many days they will stay, how many hours they want to spend each day, what they are interested in, and how they prefer to travel. Based on those choices, the system creates a clear day by day plan.

Instead of only showing locations on a map, the application organizes them into a real itinerary. It arranges places in a logical order, connects them with routes, and keeps travel time as low as possible. The user can also define where each day starts and ends, such as a hotel or another chosen location, so the plan feels more personal and realistic.

The final result works both digitally and in print. The user can view the itinerary on a map, follow the route, and print either each day separately or the full trip together.

Scope

This project focuses only on Washington, DC so the work can go deeper instead of becoming too broad. The goal is to build one strong example that shows how geospatial tools can support real decisions.

The system allows the user to choose:

  • How many days they want to stay
  • How many hours they want to spend each day
  • Their interests such as museums, parks, history, monuments, and food
  • Whether food stops should be included
  • Whether they prefer walking, public transportation, or driving
  • Where each day should start and end, such as a hotel or selected location

Based on these choices, the system builds a plan where each day includes a sequence of places that make sense geographically and are arranged in order.

Project Idea

The main idea is to create a smart itinerary planner that respects both space and time. If someone is staying for three days, wants to spend about six hours each day, is interested in history and parks, and prefers walking or public transportation, the system should respond with a plan that actually feels usable.

Each day includes a reasonable number of stops. The locations are grouped so they are close to each other. The stops are ordered clearly as 1, 2, 3, and so on. Routes connect the stops so the user understands how to move from one place to the next. The plan should feel natural and avoid sending the user back and forth across the city.

Data Sources

OpenStreetMap

This is the main source for attractions and supporting map data. It provides museums, monuments, parks, restaurants, cafes, roads, metro stations, and paths. It gives the project a strong spatial foundation and also supports route generation.

Wikipedia and Wikivoyage

Wikipedia adds short descriptions for important places. Wikivoyage adds travel-oriented context, especially at the neighborhood level, helping connect individual locations into a more complete travel experience.

Transportation and routing data also come from open sources. Road and path networks from OpenStreetMap can be used to generate routes between stops. Public transportation can be approximated using available open transit data, depending on how detailed the implementation becomes.

Process

The process starts with collecting data from open sources and preparing it into a clean and consistent dataset. After that, the focus shifts to understanding how locations relate to each other.

The most important question guiding the work is simple: What is the best order to visit these places while minimizing travel time and staying within the user’s daily limits? Locations are grouped based on proximity, filtered based on user interests, and organized into daily plans. The number of stops is adjusted based on how much time the user wants to spend each day.

In the end, the system produces a structured itinerary that can be explored on a map and also printed.

Technical Architecture

The project follows a complete but simple workflow. Open data sources provide the raw information. Python is used to clean the data and apply the logic for ranking and organizing locations. QGIS is used for spatial analysis, clustering, route visualization, and map design. A spatial database is used to store and manage the data. A lightweight web application allows users to interact with the final result.

Routing between stops is supported using open road network data, which makes it possible to generate paths that follow real streets and paths.

Advanced QGIS Analysis and Visualization

QGIS is the core of this project and where most of the value is created. One of the most important steps is grouping nearby locations. This helps identify areas where a visitor can spend time without traveling far. These clusters often match real-world areas like the National Mall or Georgetown.

After grouping, the next step is ordering the locations in a way that makes sense. The goal is to move from one nearby place to the next instead of jumping across the city. Accessibility is also considered by looking at how close locations are to transit or walkable routes. Density patterns help explain why some areas are prioritized.

One of the most interesting parts of the project is how time is handled. Locations are not just grouped in space, but also assigned to specific days. The map clearly shows numbered stops in order, routes connecting those stops, and different location types styled in a way that is easy to understand.

The project also produces printable maps. Each day can be printed separately with its own route and numbered stops, and the full itinerary can also be printed as one combined map.

Implementation Phases

The project starts with collecting open data for attractions, food, parks, roads, and transportation features in Washington, DC. The next step is cleaning and organizing the data into a consistent structure with clear categories.

After that, spatial analysis is performed in QGIS to group locations, understand proximity, and support route-based planning. Then the itinerary is created based on user inputs such as trip length, daily time, interests, travel mode, and start and end points.

Once the plan is ready, maps are designed to clearly show the ordered stops and routes. Finally, a simple web application is built to allow users to interact with the plan and view it before printing.

Key Features

  • The user chooses how many days they want to stay and how many hours they want to spend each day
  • The user selects their interests and decides whether food should be included
  • The user chooses how they want to travel and defines where each day starts and ends
  • The system generates a day by day plan where locations are grouped and ordered to reduce unnecessary travel
  • Each stop is shown on the map with a number so the order is clear
  • Routes connect the stops so the user can follow the path easily
  • The plan can be viewed online and printed either day by day or as a full itinerary

Conclusion

This project shows how geospatial tools can be used to solve a real and familiar problem in a practical way. It brings together open data, spatial analysis, and clear map design to create something that is easy to understand and useful.

More importantly, it shows that GIS is not just about displaying data. It is about using location to guide decisions, organize experiences, and create something that people can actually use in real life.
